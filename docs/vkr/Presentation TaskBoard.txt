Слайд 2:
Актуальность темы
Небольшим командам и учебным группам требуется учёт задач и проектов без обязательной привязки к внешнему облаку. Коммерческие сервисы нередко стоят денег или содержат функции, которые для учебной постановки не нужны; размещение данных у третьей стороны тоже не всегда желательно. Развёртывание у себя снимает часть этих ограничений. В качестве объекта ВКР удобно показать разбиение на сервисы, защиту HTTP-API и прогон тестов в среде непрерывной интеграции.

Слайд 3:
Цель и задачи работы
Цель - работоспособный прототип: проекты и задачи, вход, роли, уведомления, вложения, отчёты
Задачи:
- микросервисы, у каждого своя БД
- backend Java/Spring Boot, клиент React
- файлы в MinIO, учёт в PostgreSQL
- тесты Maven, Vitest, Playwright; CI в GitHub Actions

Слайд 4:
Ограничения задачи
Рассматривается учебный прототип, а не готовый коммерческий продукт. Сборка и запуск рассчитаны на Docker Compose. Обмен между сервисами идёт по HTTP; отдельный внутренний вызов для записи уведомлений закрыт ключом в заголовке запроса. В работе делается упор на согласованность архитектуры, проверки прав на стороне сервера и прохождение основных сценариев в интерфейсе.

Слайд 9:
Состав backend-сервисов
Сервис | База данных | Назначение
auth-service | taskboard_auth | Регистрация, вход, выдача и проверка JWT, профиль, список пользователей, смена пароля; смена ролей для учётной записи администратора
tasks-service | taskboard_tasks | Проекты, состав участников, задачи, комментарии, вложения (файл в MinIO, запись в БД), журнал изменений; ограничения по роли и участию в проекте
analytics-service | taskboard_analytics | Уведомления пользователю, поток SSE, сводные отчёты и выгрузка в CSV
Общий модуль taskboard-common — общая библиотека разбора и проверки JWT (в терминах Spring Boot — конфигурация OAuth2 resource server) для tasks-service и analytics-service

Слайд 12:
Описание программной разработки
Серверная часть разбита на три приложения с разными схемами БД. Браузер ходит в REST-API и передаёт JWT. Правила «кто что может менять» задаются в классах сервиса, а не в контроллере; для проектов разведены операции чтения и изменения. Уведомление из tasks-service в analytics-service передаётся отдельным запросом с секретным ключом в заголовке. Содержимое файлов вложений лежит в MinIO.

Слайд 14:
Тестирование
Backend: тесты Maven в проектах auth-service, tasks-service, analytics-service
Клиент: модульные тесты на Vitest
Сквозная проверка: Playwright по уже поднятому docker compose

На машине разработчика: сценарий run-tests.ps1, в т.ч. режимы только backend или только e2e

Тестирование (продолжение)

Слайд 16:
Результат
Собран прототип микросервисной системы учёта задач с веб-интерфейсом: роли, уведомления, вложения, отчёты. Данные разделены по сервисам; запуск описан через Docker Compose. Набор тестов и задача в CI снижают риск поломки уже сделанного функционала при доработках. Работа может служить основой для защиты и для последующего расширения возможностей.
