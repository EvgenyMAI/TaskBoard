# ОТЧЕТ О ПРОДЕЛАННОЙ РАБОТЕ ПО ВКРБ

## 1. Общие сведения

**ФИО:** Кострюков Евгений Сергеевич  
**Группа:** М8О-407Б-22  
**Тема ВКРБ:** Разработка микросервисного веб-приложения для управления внутренними задачами, проектами и коммуникацией команды.

## 2. Введение

В рамках выполнения выпускной квалификационной работы разрабатывается программная система для организации командной работы: постановки задач, контроля выполнения, распределения ответственности между участниками и сопровождения коммуникации внутри проекта.

Актуальность темы обусловлена тем, что готовые SaaS-решения для управления задачами часто требуют платной подписки, зависят от внешней инфраструктуры и не всегда подходят для учебных или локальных корпоративных сценариев. Разработка собственного решения позволяет получить контролируемую архитектуру, адаптируемую под конкретные требования, а также продемонстрировать практическое применение современных подходов к построению распределенных систем.

Цель текущего этапа — создание и проверка работоспособного MVP-прототипа, включающего основные бизнес-функции, пользовательский интерфейс, механизмы безопасности и средства отладки.

## 3. Архитектура системы

Система построена по микросервисной архитектуре и включает отдельные сервисы аутентификации, задач и аналитики, а также веб-клиент. Каждый backend-сервис использует собственную базу данных PostgreSQL, а файловые вложения задач хранятся в отдельном объектном хранилище MinIO, что обеспечивает логическую изоляцию и упрощает дальнейшее масштабирование.

Взаимодействие между клиентом и бэкендом выполняется через REST API. Для авторизации используется JWT-токен. Внутренние межсервисные вызовы (например, отправка уведомлений из сервиса задач в сервис аналитики) защищены отдельным внутренним ключом.

### 3.1 Состав сервисов

| Сервис | Порт | База данных | Назначение |
|---|---:|---|---|
| `auth-service` | 8081 | `taskboard_auth` | Регистрация, вход, профиль пользователя, список пользователей |
| `tasks-service` | 8082 | `taskboard_tasks` | Проекты, задачи, комментарии, вложения, история изменений |
| `analytics-service` | 8083 | `taskboard_analytics` | Уведомления, сводные отчеты |
| `frontend` | 3000 | - | Пользовательский веб-интерфейс |

На уровне реализации backend-микросервисов контроллеры выполняют преимущественно транспортную роль (разбор запроса, извлечение идентификатора пользователя из JWT, вызов сервисов приложения), а сценарии использования и проверки прав сосредоточены в сервисном слое. Такое разделение соответствует принципу единственной ответственности и облегчает модульное тестирование. Для сервисов задач и аналитики общая библиотека `taskboard-common` унифицирует проверку JWT, выдаваемых `auth-service`, без копирования одного и того же кода в несколько модулей.

## 5. Реализованный функционал

### 5.1 Пользовательский интерфейс

Реализованы рабочие страницы:

- авторизация и регистрация;
- Dashboard;
- проекты и карточка проекта;
- задачи и карточка задачи;
- аналитика и отчёты;
- уведомления;
- профиль пользователя.

Также реализованы inline-валидация форм, toast-уведомления, адаптивная навигация и skeleton-загрузки.
Интерфейс основных разделов выдержан в едином ритме: светлые поверхности карточек и секций, согласованная типографика и акценты, верхний hero-блок с аватаром по инициалу пользователя, чипы с явными подписями показателей (число задач, проектов, непрочитанных уведомлений, выбранный период отчёта и т.д.), ниже — секции с маркером раздела. Страница аналитики использует тот же каркас: выбор интервала `from`/`to`, пресеты длительности, выгрузка CSV; в сводных показателях выделены подблоки «Контекст» (объём задач, просрочка, без исполнителя) и «Коэффициенты» (завершение, доля просрочки, доля назначенных) без дублирования детального разбора по статусам; детальное распределение по статусам вынесено в отдельную секцию с переключением вида «кольцо» или «столбцы»; при наличии данных отображается динамика периода; далее — топы проектов и исполнителей.
Записи ленты отображаются как разворачиваемые элементы с кратким превью и полным текстом при раскрытии, фильтрация в интерфейсе выполняется по прочитанности, типу и поисковой строке (без выбора периода по датам); непрочитанная запись при раскрытии автоматически помечается прочитанной, предусмотрена массовая отметка всех уведомлений прочитанными.
Карточка задачи оформлена в едином стиле со страницей списка задач: верхний блок с основными полями и действиями, ниже — переключение вкладок «Комментарии», «Вложения», «История». Комментарии и записи истории выводятся отдельными карточками; в истории подписи полей приведены к русским названиям, для изменений показаны значения «было» и «стало». Вложения отображаются адаптивной сеткой карточек: для изображений предусмотрены миниатюра и просмотр в модальном окне по клику, для текстовых файлов — встроенное чтение в карточке; высота карточки вложения не подстраивается под более высокую соседнюю (например, с превью фотографии), чтобы не оставлять пустого пространства у файлов без превью. При выборе исполнителя в форме доступны только пользователи из состава участников проекта.
Страница списка проектов и карточка выбранного проекта следуют той же схеме, что и список задач: hero, секции с фильтрами и списком задач проекта, форма создания задачи. Для администратора и менеджера управление участниками проекта вынесено в сворачиваемый блок (по умолчанию закрыт), с поиском по имени, почте или идентификатору и постраничным просмотром длинных списков; приглашение новых участников доступно внутри этого блока.
Дополнительно реализованы элементы ролевого интерфейса: управление ролями пользователей (`ADMIN`) на странице профиля в виде сворачиваемого блока с ленивой загрузкой списка, поиском по имени и адресу электронной почты и пагинацией.

### 5.2 Auth Service

Реализованы функции:

- регистрация пользователя;
- вход и выдача JWT;
- проверка JWT;
- получение и обновление профиля;
- смена пароля с валидацией;
- получение списка пользователей (для назначения задач и администрирования): в сводке передаются идентификатор, имя пользователя, адрес электронной почты и набор ролей;
- изменение ролей пользователей (`ADMIN` / `MANAGER` / `EXECUTOR`) через защищенный API-метод для администраторов.

### 5.3 Tasks Service

Реализованы функции:

- CRUD проектов;
- CRUD задач с фильтрацией по проекту, статусу и исполнителю;
- комментарии к задачам;
- вложения к задачам;
- загрузка вложений через multipart API, хранение файлов в MinIO и хранение метаданных в PostgreSQL;
- просмотр изображений и текстовых файлов в UI, скачивание любых форматов;
- история изменений задач;
- управление участниками проекта через отдельную таблицу membership (`project_members`);
- проверка доступа к проектам/задачам через membership и ролевые ограничения;
- проверка при создании и обновлении задачи: если указан исполнитель, он должен входить в состав участников проекта; при нарушении возвращается ошибка с поясняющим сообщением в теле ответа для отображения в клиенте;
- отправка внутренних уведомлений в analytics-service при событиях в задачах.

### 5.4 Analytics Service

Реализованы функции:

- хранение уведомлений в PostgreSQL;
- получение списка уведомлений с фильтрацией;
- подсчет непрочитанных уведомлений;
- отметка уведомления как прочитанного;
- API сводной отчетности (`summary`, `by-project`, `by-assignee`) с фильтрацией по периоду `from/to`;
- экспорт отчета в CSV, адаптированный для Excel (UTF-8 BOM, русские подписи, читаемый формат даты/времени);
- фронтенд-вкладка «Аналитика»: период и пресеты, CSV, сводные показатели (контекст и коэффициенты), динамика периода, распределение по статусам (кольцо или столбцы), топы проектов и исполнителей; оформление согласовано с экранами профиля и уведомлений.

## 6. Безопасность системы

На текущем этапе обеспечены следующие меры безопасности:

- настройка **CORS** для REST API сервисов задач и аналитики при работе веб-клиента с отдельного origin (режим разработки Vite);
- JWT-аутентификация для защищенных API-методов;
- хранение паролей в виде BCrypt-хеша;
- ролевая модель доступа (RBAC) с ролями `ADMIN`, `MANAGER`, `EXECUTOR`, включенными в JWT;
- централизованные проверки прав на уровне backend-контроллеров (проекты, задачи, комментарии, вложения);
- разграничение доступа к проектам через membership-модель (таблица `project_members`);
- хранение содержимого вложений в MinIO и выдача файлов только через защищенные endpoint `tasks-service` (по JWT);
- запрет доступа к уведомлениям других пользователей (фильтрация по `userId` из токена);
- защита внутренних межсервисных уведомлений через `NOTIFICATIONS_INTERNAL_KEY`;
- принудительная привязка `userId` к аутентифицированному пользователю для публичных операций создания уведомлений.

## 7. Логирование и отладка

Для сопровождения и диагностики реализовано централизованное file-based логирование backend-сервисов с ротацией логов. В логах фиксируются:

- идентификатор запроса (`requestId`);
- идентификатор пользователя (`userId`, если доступен);
- HTTP-метод, endpoint, статус ответа и время выполнения.

Логи сервисов доступны в отдельных файлах:

- `logs/auth-service/auth-service.log`;
- `logs/tasks-service/tasks-service.log`;
- `logs/analytics-service/analytics-service.log`.

Это позволяет проводить анализ поведения системы без подключения к контейнерам.

## 8. Запуск системы

Для локального запуска используется **Docker Compose** (из корня репозитория) и **Vite** для фронтенда.

**Минимальная последовательность на новой машине** (после установки Docker и Node.js, клонирования репозитория):

1. Поднять backend и базы данных:

   `docker compose up -d --build`

   При необходимости явно зафиксировать порты хоста (8081 / 8082 / 8083 и др.) без зависимости от переменных окружения в текущей сессии терминала:

   `docker compose --env-file docker.defaults.env up -d --build`

2. В каталоге `frontend` выполнить `npm install` и `npm run dev`.

3. Веб-интерфейс: `http://localhost:3000`. Первый зарегистрированный пользователь получает роль администратора.

Отдельное копирование `frontend/.env` для разработки **не требуется**: в репозитории закоммичен **`frontend/.env.development`** с URL API на `localhost:8081`, `8082`, `8083`, что соответствует портам по умолчанию в `docker-compose.yml`. При нетипичных портах используются файлы `.env` в корне и/или `frontend/.env.local` (см. README, раздел 9).

Подробная инструкция по окружениям, пересборке образов, очистке артефактов и тестам приведена в **README.md (раздел 9)** в конце файла.

## 10. Текущее состояние проекта

На текущий момент реализован и проверен рабочий MVP-прототип системы. Поддерживается полный базовый цикл работы с проектами и задачами через веб-интерфейс: от регистрации пользователя до постановки задач, изменения статусов, ведения истории изменений и получения уведомлений.
Реализована и проверена базовая ролевая модель и сценарии командной работы: администрирование ролей, управление участниками проекта и контроль прав на операции с задачами.

Архитектура решения является расширяемой: сервисы изолированы, API-границы определены, инфраструктура развертывания стандартизирована через Docker Compose, что позволяет переходить к следующему этапу развития без архитектурного пересмотра ядра системы.

**Дополнения по инфраструктуре и сборке (после выделения общего модуля и доработок сопровождения):** корневой Maven-reactor объединяет `taskboard-common`, `tasks-service` и `analytics-service`; Docker-образы задач и аналитики собираются с контекстом корня репозитория, пути `COPY` сохраняют каталог `src/main/java` внутри образа. Для стабильной работы веб-клиента с отдельных origin в `tasks-service` и `analytics-service` в конфигурации Spring Security включена поддержка CORS совместно с JWT-фильтрами. Для разработчика добавлены файлы `docker.defaults.env`, `frontend/.env.development`, скрипт очистки артефактов `scripts/clean-dev.ps1`; скрипт `run-tests.ps1` доработан так, чтобы локальный e2e не оставлял случайные порты в переменных процесса терминала и чтобы итог прогона backend отражал реальный код выхода и суммарное число тестов по модулям.

## 11. Автоматизированное тестирование и CI/CD

На текущем этапе реализована многоуровневая стратегия тестирования, ориентированная на снижение регрессий и сокращение объема ручной проверки:

1. **Integration-тесты backend-сервисов** (`auth-service`, `tasks-service`, `analytics-service`);
2. **Модульные (unit) тесты** отдельных сервисов и тонких контроллеров (JUnit 5 + Mockito), в том числе для логики уведомлений, выборки задач с учётом роли исполнителя и расчёта показателей отчётов;
3. **E2E smoke-тест** ключевого пользовательского сценария через Playwright;
4. **CI pipeline** с автоматическим запуском тестов в GitHub Actions.

### 11.1 Integration-тесты backend

Интеграционные тесты построены на Spring Boot Test + MockMvc и запускаются в профиле `test` с in-memory БД H2 (`application-test.yml` в каждом сервисе). Такой подход позволяет:

- изолировать тестовые данные от рабочих PostgreSQL-контуров;
- проверять бизнес-правила и API-контракты на уровне реальных контроллеров/сервисов;
- быстро воспроизводить критичные сценарии (`RBAC`, membership, уведомления, вложения, авторизация).

Unit-тесты дополняют интеграционные проверки: они нацелены на конкретные ветки правил (например, ограничение списка задач для исполнителя, нормализация параметров уведомлений, агрегирование метрик) без полного подъёма контекста Spring и работают быстрее полного прогона интеграционного теста.

### 11.2 E2E smoke-тест

E2E-сценарий реализован в `frontend/tests-e2e/smoke.spec.js` и покрывает сквозной пользовательский поток:

- создание проекта через UI;
- подготовка данных (участник/задача) через API для повышения стабильности smoke;
- проверка доставки уведомления исполнителю в UI;
- открытие страницы «Аналитика» и наличие ключевых разделов интерфейса (заголовок, блок периода и выгрузки, сводные показатели);
- отдельный шаг Playwright **setup** сохраняет сессию первого пользователя (ADMIN в пустой БД) для сценариев, требующих прав менеджера;
- дополнительные сценарии: дашборд и навигация, страницы «Задачи», «Профиль», «Уведомления», список проектов и карточка проекта после создания через API.

### 11.3 Локальный запуск и developer workflow

Для разработчиков реализован единый скрипт `run-tests.ps1`:

- `-BackendOnly` — только integration backend;
- `-E2EOnly` — только e2e;
- `-VerboseBackendLogs` — подробный Maven-лог backend в консоль;
- `-UseHostMaven` — backend-тесты через локальный `mvn` (JDK 17) с тем же локальным репозиторием под `.cache/m2/repository`;
- без параметров — полный прогон.

Скрипт автоматически поднимает изолированный временный compose-стек для e2e, подбирает свободные порты через **временный файл `--env-file`**, по завершении восстанавливает ранее заданные в сессии переменные портов и выполняет cleanup (`docker compose down -v`), что исключает накопление временных контейнеров/томов и **не ломает** последующий ручной запуск `docker compose` / `npm run dev` на стандартных портах.

Для backend-части используется общий Maven cache (`.cache/m2`), что сокращает повторные скачивания зависимостей при последующих прогонах. По умолчанию запуск Maven выполняется через `docker`; опционально — на хосте (`-UseHostMaven`). Фиксируется **код выхода**; в логе дополнительно проверяются строка `BUILD FAILURE` и **сумма тестов** по блокам Surefire `[INFO] Results:` (в т.ч. для одного объединённого прогона `tasks-service` и `analytics-service`), чтобы не получить ложноположительный «успех» при сбое.

По завершении backend-тестов выводится краткая сводка (`Backend summary`) и итоговый KPI-блок (`Backend KPI`, в т.ч. суммарное число тестов по Results), а полный лог каждого сервиса сохраняется в `logs/test-runs/<timestamp>/`. После этапа e2e выполняется проверка, что контейнеры `tasks-service` и `analytics-service` остаются в состоянии `running`.

E2E-проверяет согласованность портов **внутри своего временного стека** и не заменяет ручную проверку согласования `frontend/.env.development` с опубликованными портами Docker при повседневной разработке (см. README, раздел 9).

### 11.4 CI/CD

Файл `.github/workflows/ci.yml` запускает:

- backend-тесты на каждом `push` и `pull_request`;
- e2e smoke после успешного backend-джоба.

Таким образом, тесты встроены в ежедневный процесс разработки и выполняют роль обязательного quality gate перед слиянием изменений.

## Приложение А. Ключевые API-методы

### Auth Service

- `POST /api/auth/register` — регистрация пользователя;
- `POST /api/auth/login` — вход и получение JWT;
- `GET /api/auth/me` — получение профиля;
- `PUT /api/auth/me` — обновление профиля;
- `PUT /api/auth/me/password` — смена пароля;
- `GET /api/users` — список пользователей (`id`, `username`, `email`, `roles`);
- `PUT /api/auth/users/{id}/roles` — изменение роли пользователя (для администратора).

### Tasks Service

- `GET /api/projects`, `POST /api/projects`, `PUT /api/projects/{id}`, `DELETE /api/projects/{id}`;
- `GET /api/projects/{id}/members`, `POST /api/projects/{id}/members`, `DELETE /api/projects/{id}/members/{userId}`;
- `GET /api/tasks`, `POST /api/tasks`, `PUT /api/tasks/{id}`, `DELETE /api/tasks/{id}`;
- `GET /api/tasks/{taskId}/comments`, `POST /api/tasks/{taskId}/comments`;
- `GET /api/tasks/{taskId}/attachments`, `POST /api/tasks/{taskId}/attachments/upload`, `GET /api/tasks/{taskId}/attachments/{id}/download`, `GET /api/tasks/{taskId}/attachments/{id}/preview`, `DELETE /api/tasks/{taskId}/attachments/{id}`;
- `GET /api/tasks/{taskId}/history?limit=50`.

### Analytics Service

- `GET /api/notifications/stream` (SSE) — поток realtime-уведомлений для текущего пользователя;
- `GET /api/notifications` — список уведомлений (на стороне API поддерживаются параметры `read`, `type`, период `from`/`to`, поиск `q`, `limit`; в веб-интерфейсе используются фильтры без периода по датам);
- `GET /api/notifications/unread` — количество непрочитанных;
- `PATCH /api/notifications/{id}/read` — отметить как прочитанное;
- `GET /api/reports/summary` — краткая сводка;
- `GET /api/reports/by-project` — агрегация по проектам;
- `GET /api/reports/by-assignee` — агрегация по исполнителям;
- `GET /api/reports/export` — экспорт отчета в CSV (Excel-совместимый формат).
